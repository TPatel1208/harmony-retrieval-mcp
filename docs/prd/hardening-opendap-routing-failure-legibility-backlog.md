# PRD: Hardening plan — OPeNDAP quirk containment, routing legibility, uniform failure legibility, and triage-backlog closure

> Triage label to apply when published: `ready-for-agent`
> Umbrella plan: four workstreams, each independently landable as its own session/commit per project discipline. Do not implement more than one workstream per session.

## Problem Statement

Four recurring pain areas, each confirmed against the current code and commit history:

**1. OPeNDAP/DAP4 correctness is the most-churned area and its knowledge is trapped in commit messages.** The OPeNDAP provider module is the largest source file and its test file the largest test file. The history shows the same shape of fix landing repeatedly: coordinate-aware DAP4 constraint expressions, grouped-netCDF4 fully-qualified names (leading slash — Hyrax 400s without it), bbox→hyperslab index translation (UMM-C bbox is grid-edge, not cell-center), CF-time decoding before time-range selection, parallelization, group-flattened lazy reads. Each fix is a reverse-engineered Hyrax/DAAC quirk, and each one risks regressing a previous one because the quirks are pinned by scattered per-bug tests rather than a single regression corpus. From a maintainer's perspective: "every new collection archetype is a fresh 400, and I can't tell which past quirks my fix might un-fix."

**2. Harmony service selection works but is illegible.** The union-trap is correctly handled (`find_service` matches one whole service or returns `None`; the rolled-up booleans are never read; no-match submits unpinned). But the *decision* is invisible: a researcher whose job went unpinned — or whose pinned service was chosen over three others — has no durable record of why. Because service registrations across DAACs are heterogeneous and change out from under us, "did the gate decide correctly for this collection" is a question we currently answer by re-reading code, not by reading the job. The fixture base for `find_service` is also thin relative to the heterogeneity it must handle (one real union-trap fixture; no no-services, no multi-service-partial-overlap corpus).

**3. Failure legibility is being bought one PRD at a time.** The worker's `submit_job` now explains the no-OPeNDAP-fallback case well (the `OPENDAP_NOT_APPLICABLE` event + prefixed error), but that legibility was a one-off patch: `poll_job` and `materialize_job` still store a bare exception message with no stage, no provider, and no provenance event. A researcher looking at a `FAILED` job cannot tell from the durable record whether the failure happened at submit, poll, or materialize, nor through which provider — the exact "correct behavior that looks like a bug" problem the last two PRDs each solved for one branch. The recurring design tax is the absence of a convention: there is no single choke point through which every terminal failure passes.

**4. The deferred triage backlog is going stale unevenly.** Of the seven deferred items from the last triage round, two are already fixed in the meantime (`preview_dataset` now does an authoritative CMR EDSC-tag lookup with the short-name guess only as last resort; the job-not-found error already renders as `unknown job: '<handle>'`). The five still open: `estimate_retrieval_size` computes a total but never warns when it is enormous, despite its own docstring calling it "the gate a later phase uses to refuse a huge request before any byte moves"; `retrieve_timeseries(point_sample=True)` mints a durable job and *then* discovers (inside the provider's submit) that the product isn't in the AppEEARS catalog, so the researcher gets a job handle for a job that can never succeed; `define_area_of_interest` echoes the full geocoded GeoJSON geometry (Nominatim multipolygons can be enormous) in the tool response; `describe_dataset` returns every variable fully enriched plus the whole collection metadata blob, which for large collections dwarfs what an agent needs; and `check_coverage`'s `sample_granules` duplicates `inspect_granules`' full granule records instead of being the eyeball list its docstring describes.

## Solution

Four workstreams, in the project's own vocabulary (Harmony-first, union-trap, durable job, re-materializable spec, handles):

**WS1 — Contain the Hyrax quirks in one pure, golden-tested core.** Separate the DAP4 constraint-expression/hyperslab serialization into a pure, public, I/O-free core with a table-driven **collection-archetype corpus**: each archetype (flat root-level à la GLDAS, grouped à la TEMPO, names-with-spaces à la MOD13Q1, grid-edge-bbox à la the UMM-C convention) is a named fixture carrying its UMM-V/geometry shape and the exact expected constraint-expression string. Every quirk fixed so far becomes a corpus row; every future quirk becomes a new row or archetype instead of an ad-hoc test. Pair it with a Hyrax-quirk ledger document where each entry cross-references the corpus row that pins it, so the reverse-engineered knowledge survives outside commit messages. Public interface (`plan_subset`, the provider class) unchanged.

**WS2 — Make the routing decision durable and explainable.** `route`/`find_service` behavior is unchanged; what changes is that the decision now says why. The routing decision carries a trace (which service pinned, or per-service the first unmet need when none matched), a `ROUTED` provenance event records that trace at plan time, and the `find_service` test corpus grows into a recorded-fixture set of real heterogeneous DAAC registrations (union-trap, no-services, partial-overlap multi-service) so gate correctness is asserted against reality, not synthetic minimal fixtures.

**WS3 — One failure envelope for every terminal failure.** A single helper through which all three lifecycle tasks (submit, poll, materialize) pass on their way to `FAILED`: it builds the stored error string from stage + provider + exception, and records a `JOB_FAILED` provenance event whose detail carries the same structured facts — using only information already on the durable spec or in the caught exception, never re-derived at failure time. The existing `OPENDAP_NOT_APPLICABLE` path becomes an instance of the convention rather than an exception to it. The rule itself ("anything needed to explain a failure must already be on the spec at plan time") is codified in the domain glossary so future failure modes get legibility by construction.

**WS4 — Close the five still-open triage items; record the two already-fixed ones as done.** Size-threshold warning on `estimate_retrieval_size`; planning-time AppEEARS catalog check for point-sample plans (fail fast before any job/handle is minted); slim `define_area_of_interest`'s response (geometry summary, not the full GeoJSON — the durable payload keeps the geometry); a compact-by-default `describe_dataset` with opt-in full detail; and a slim eyeball projection for `check_coverage`'s sample so it stops duplicating `inspect_granules`.

## User Stories

### WS1 — OPeNDAP quirk containment

1. As a maintainer, I want every Hyrax/DAP4 quirk we have reverse-engineered (grouped FQN leading slash, coordinate-aware hyperslabs, grid-edge bbox translation, CF-time decode-before-subset) pinned by one named row in a single archetype corpus, so that I can see at a glance which quirks exist and know a change that breaks one fails a named test, not a mystery test.
2. As a maintainer fixing the next per-DAAC quirk, I want to add a corpus row (or a new archetype fixture) rather than invent a new test style, so that the corpus grows monotonically and the module's regression surface stays in one place.
3. As a maintainer, I want the constraint-expression/hyperslab serialization to be a pure function of (resolved variables, axis geometry, bbox) with no network or file I/O, so that the corpus tests are exact string assertions that run instantly and can never flake.
4. As a maintainer, I want the OPeNDAP module decomposed along its three existing responsibilities — pure DAP4 serialization, CMR-facing planning (`plan_subset` and its discovery helpers), and Hyrax-facing runtime (the provider's submit/poll/materialize) — so that reading one concern no longer requires scrolling past the other two in an 800-line file.
5. As a maintainer, I want the module's public face (`plan_subset`, the provider class, the plan dataclass) unchanged by the decomposition, so that the retrieval tool, `providers.build`, and the worker are untouched and the change is invisible outside the module.
6. As a maintainer, I want a Hyrax-quirk ledger document where each entry states the quirk, the DAAC/collection it was observed on, and the corpus row that pins it, so that the knowledge currently trapped in commit messages is readable in one place.
7. As an Earthdata MCP user, I want retrievals against flat, grouped, and names-with-spaces collections to behave exactly as they do today after the decomposition, so that this workstream is pure containment with zero functional change.
8. As a maintainer, I want the existing scattered per-bug tests preserved or absorbed into the corpus without losing any assertion, so that containment never trades away coverage.

### WS2 — Routing legibility

9. As a researcher, I want the durable record of my job to say which Harmony service was pinned and why — or that it was submitted unpinned because no single service satisfied the plan — so that I can tell a deliberate unpinned submission apart from a routing bug.
10. As a researcher whose plan matched no single service, I want the routing trace to name, per service, the first plan need that service failed to satisfy, so that I can relax the request ("drop reproject and service X matches") without reading UMM-S records myself.
11. As a maintainer, I want a `ROUTED` provenance event recorded at plan time carrying that trace, so that the decision is on the same durable, re-materializable record as every other lifecycle fact — never re-derived after the fact.
12. As a maintainer, I want the routing decision's trace computed inside the existing pure `route` call from data already in hand, so that legibility adds zero network calls and `route` remains a pure planning decision that never submits.
13. As a maintainer, I want a recorded fixture corpus of real heterogeneous DAAC service registrations — the TEMPO L2 union-trap, an LPCLOUD-style no-registered-services collection, and a multi-service partial-overlap case — so that `find_service` is tested against the heterogeneity it actually faces.
14. As a maintainer, I want an explicit regression test asserting the rolled-up top-level capability booleans are never consulted (the union-trap fixture advertises a union no single service can do, and the gate still returns no match), so that the project's hardest-won routing rule is pinned by a test, not only by a docstring.
15. As an Earthdata MCP user, I want every routing outcome (pinned, unpinned, direct-S3, AppEEARS, not-retrievable) to behave exactly as today, so that this workstream adds visibility without changing a single decision.

### WS3 — Uniform failure legibility

16. As a researcher looking at a `FAILED` job, I want the stored error to say which lifecycle stage failed (submit, poll, or materialize) and through which provider, so that "Harmony poll died" and "OPeNDAP materialize died" stop reading identically.
17. As a researcher, I want every terminal failure to leave a `JOB_FAILED` provenance event with structured detail (stage, provider, error type, message), so that `get_provenance` tells the full story of an unhappy job the same way it already tells the story of a happy one.
18. As a maintainer, I want all three lifecycle tasks to reach `FAILED` through one shared failure-envelope helper, so that the next failure-legibility requirement is a change to one function, not a fourth PRD.
19. As a maintainer, I want the helper to use only facts already on the durable spec or in the caught exception — never a fresh CMR/Harmony call at failure time, so that a failing job never costs an extra round-trip on top of the failure it is reporting.
20. As a researcher, I want the existing `OPENDAP_NOT_APPLICABLE` behavior (event + prefixed error) preserved verbatim as an instance of the new convention, so that the legibility already shipped is not regressed by generalizing it.
21. As a maintainer, I want the convention itself ("explanation facts are persisted at plan time; failure time only formats them") written into the project's domain glossary, so that future contributors inherit the rule and not just its instances.
22. As a researcher, I want `get_retrieval_status` on any failed job to surface the stage-and-provider-prefixed error inline, so that I don't have to cross-reference provenance to learn where a job died.

### WS4 — Triage-backlog closure

23. As a researcher, I want `estimate_retrieval_size` to attach an explicit warning when the extrapolated total exceeds a configurable threshold, so that I get "this is ~48 GB — narrow the AOI, time window, or variables" before any byte moves, matching the gate its own docstring promises.
24. As a researcher calling `retrieve_timeseries(point_sample=True)`, I want the AppEEARS product-catalog check to run at planning time and fail fast with the existing clear not-in-catalog message, so that I never receive a job handle for a job that can never succeed.
25. As a maintainer, I want that planning-time catalog check to reuse the provider's existing product lookup, run only for point-sample plans routed to AppEEARS, and cache per product, so that fail-fast costs nothing for every other plan shape.
26. As an agent calling `define_area_of_interest` for a large watershed or country, I want the tool response to carry the bbox, source, and a geometry summary (type and vertex count) instead of the full GeoJSON, so that one AOI definition doesn't flood my context window — while the durable handle payload keeps the full geometry for re-materialization.
27. As an agent calling `describe_dataset`, I want a compact default response (per-variable name, long name, units, and advisory flags; core collection metadata only) with an opt-in full-detail mode, so that describing a 400-variable collection stays usable and today's full shape stays reachable.
28. As an agent calling `check_coverage`, I want `sample_granules` to be a slim eyeball projection (granule identifier and temporal extent), so that the tool stops duplicating `inspect_granules`' full records and each of the four coverage tools answers a distinct question.
29. As a maintainer, I want the two already-fixed triage items (authoritative GIBS layer lookup; readable job-not-found error) confirmed by their existing tests and recorded as closed in this PRD, so that the backlog list stops carrying dead entries.

## Implementation Decisions

### WS1 — OPeNDAP quirk containment

- **Decompose, don't rewrite.** The OPeNDAP provider module is split along its three existing responsibilities: pure DAP4 serialization (fully-qualified names, hyperslab brackets, the constraint-expression builder, the index-range math), CMR-facing planning (the public `plan_subset` and its discovery/resolution helpers), and Hyrax-facing runtime (the provider class). The package's public exports are unchanged; nothing outside the module changes an import of a public name.
- **The serialization core becomes public and pure.** The constraint-expression builder (today a private function already tested directly) is promoted to the serialization core's public face. It takes resolved inputs (variable paths, axis geometry, bbox) and returns strings; it performs no I/O. This is the only interface promotion in the workstream.
- **Archetype corpus as data, not code.** Each archetype is a named fixture (flat/root-level, grouped, names-with-spaces, grid-edge-bbox geometry; more added as quirks arrive) bundling the CMR variable shapes, axis geometry, requested variables/bbox, and the exact expected constraint-expression string(s). Corpus tests are parametrized over the rows. Existing per-bug assertions are absorbed as rows or kept verbatim — no assertion is deleted.
- **Quirk ledger.** A documentation page lists each reverse-engineered quirk (what Hyrax does, which DAAC/collection surfaced it, what the correct emission is) and names the corpus row pinning it. New quirks require a ledger entry plus a corpus row, stated as the module's contribution convention in its docstring.
- **CF-time decode-before-subset and grouped lazy-read flattening stay where they are** (the read/concatenation path), pinned by their existing tests; the corpus covers request *emission*, not response decoding. No behavior change anywhere in this workstream.

### WS2 — Routing legibility

- **Trace on the decision, computed in `route`.** The routing-decision value gains a trace: for a pinned service, its name/concept id and the plan needs it satisfied; for an unpinned submission, a per-service list of (service name, first unmet plan need) plus the reason category (union-trap/no-single-service vs. no-registered-services); for direct-S3 and AppEEARS, the shortcut taken and its gate facts (in-region + enabled; point-sample intent). Computed from the capabilities object already in hand — `route` stays pure and never submits.
- **New `ROUTED` provenance event, no migration.** Added to the provenance event vocabulary (a plain string column, same as the prior `SUBMITTED`/`OPENDAP_NOT_APPLICABLE` additions). Recorded by the shared retrieval planning core immediately after routing, with the trace as the event detail. The durable request spec is unchanged — the trace explains a decision; the spec remains the re-materializable request.
- **Fixture corpus from recorded registrations.** The capabilities fixtures grow from synthetic minimal cases to recorded real-shape registrations: the existing TEMPO L2 union-trap record, a no-registered-services collection (the LPCLOUD case from the fallback-signal PRD), and a multi-service record where different services each satisfy different plan subsets. Fixtures are checked-in recorded JSON, never live calls.
- **No routing behavior changes.** `find_service`'s one-whole-service-or-None contract, Harmony-first, unpinned-on-no-match, the direct-S3 gate, and the AppEEARS point-sample gate are all unchanged and re-asserted by the grown corpus.

### WS3 — Uniform failure legibility

- **One failure-envelope helper in the worker.** All three lifecycle tasks route their exception handlers through it. It (a) formats the stored error string as stage- and provider-prefixed human-readable text (preserving the existing multi-arg-exception join and, on the no-fallback branch, the existing "Harmony failed and no OPeNDAP fallback is available" prefix verbatim), and (b) records a `JOB_FAILED` provenance event with detail `{stage, provider, error_type, message}` plus any stage-specific facts already at hand.
- **New `JOB_FAILED` event type, no migration** (string column, as above). `OPENDAP_NOT_APPLICABLE` continues to be recorded on its branch *in addition to* `JOB_FAILED` — it explains the fallback decision; `JOB_FAILED` records the terminal outcome. The `PROVIDER_FALLBACK` → re-enqueue path records no `JOB_FAILED` (the job is not failing; it is falling back).
- **No re-derivation at failure time.** The helper's inputs are the spec, the stage name, and the caught exception — nothing else. This is the convention, recorded in the domain glossary alongside the existing glossary entries.
- **`get_retrieval_status` needs no new code** — it already surfaces the stored error string; the prefix convention makes that string self-explanatory, same mechanism as the fallback-signal PRD.

### WS4 — Triage-backlog closure

- **Size-threshold warning:** a settings value (size-warning threshold, default on the order of ~10 GB) checked by `estimate_retrieval_size` after extrapolation; when exceeded, the existing `warning` field carries a message stating the estimated total and suggesting a narrower AOI, shorter time window, or variable subsetting. Estimation math and response shape otherwise unchanged; the existing no-granules / no-sizes warnings keep priority (they are mutually exclusive with a large estimate).
- **Planning-time AppEEARS catalog check:** the shared retrieval planning core, only on the point-sample path that routes to AppEEARS, performs the provider's existing product-catalog lookup before any job row or handle is minted; a not-in-catalog product raises a `ValueError` carrying the provider's existing clear message. Result cached per product identifier for the process lifetime. The provider's own submit-time check stays (the worker must remain safe for resumed jobs).
- **AOI response slimming:** `define_area_of_interest` keeps persisting the full geometry in the durable handle payload (provenance/re-materialization unchanged) but its tool response replaces the echoed GeoJSON with `{geometry_type, vertex_count}` alongside the existing bbox/source/query fields. Consumers needing the geometry resolve the handle.
- **Compact `describe_dataset`:** default response trims each variable to name, long name, units, and its advisory notes, and trims the collection metadata echo to core identification fields; a new opt-in detail parameter returns today's full shape. The variables list also reports a total count so truncation is never silent.
- **Slim coverage sample:** `check_coverage.sample_granules` becomes a projection of granule identifier + temporal extent; `granule_count`/`covered`/`sampled_granules` semantics unchanged; `inspect_granules` remains the tool for full records.
- **Closed without work:** the GIBS authoritative lookup (CMR EDSC-tag with short-name guess as documented last resort) and the readable job-not-found message are already implemented and tested; this PRD records them as done.

### Sequencing

Each workstream is one session, one commit. WS4's five items are small enough to land as one session together, or individually. Recommended order: WS3 (smallest, establishes the failure convention) → WS2 (adds `ROUTED` on the same provenance machinery) → WS1 (largest, pure containment) → WS4 anytime (fully independent). No workstream depends on another; the order is only a suggestion.

## Testing Decisions

Good tests here assert external, observable behavior at existing seams — emitted constraint-expression strings, routing decisions returned, durable provenance events and stored error strings, tool response shapes — never internal call sequences or mock call counts. No new seams are introduced anywhere in this plan; one existing seam is promoted (the constraint-expression builder becomes public), which formalizes where its tests already run.

- **WS1 — the pure serialization core (existing seam, promoted).** The archetype corpus runs as parametrized exact-string tests against the promoted constraint-expression builder; `plan_subset` keeps its existing fake-CMR unit tests (bare-leaf resolution, ambiguity raise, fail-soft on CMR error, geometry discovery). Prior art: the existing constraint-expression and `plan_subset` coverage in `tests/unit/test_providers/test_opendap.py`. The retrieval-tool tests in `tests/unit/test_tools/test_retrieval.py` must pass unchanged — they pin that the decomposition is invisible upstream.
- **WS2 — `Router.route` / `find_service` (existing seams).** Corpus-driven tests in `tests/unit/test_router.py` and `tests/unit/test_capabilities.py` over the recorded registration fixtures: pinned trace names the service; unpinned trace lists per-service first-unmet-need; union-trap fixture yields no match with the rolled-up booleans untouched. The `ROUTED` event is asserted at the durable-record seam the same way `SUBMITTED` is today (prior art: `tests/unit/test_tools/test_retrieval.py` provenance assertions).
- **WS3 — the worker lifecycle tasks via the durable record (existing seam).** For each stage (submit/poll/materialize), a forced provider exception yields: a `FAILED` job whose error string carries the stage+provider prefix and the original message; exactly one `JOB_FAILED` event with the structured detail; the `OPENDAP_NOT_APPLICABLE` and `PROVIDER_FALLBACK` branches unchanged (existing tests keep passing). Prior art: `tests/unit/test_jobs/test_worker_provenance.py`.
- **WS4 — the tool functions (existing seams).** `estimate_retrieval_size` over/under threshold → warning present/absent with unchanged math; point-sample against a fake AppEEARS 404 → `ValueError` raised and *no* job row or handle created (assert absence at the store/DB, not on mocks); AOI response carries summary not geometry while the handle payload still holds the full geometry; `describe_dataset` compact vs. full detail shapes; `check_coverage` slim sample with unchanged counts. Prior art: `tests/unit/test_tools/test_coverage.py`, `test_area.py`, `test_discovery.py`, `test_retrieval.py`.
- **Run everything via the project's Docker test workflow** (`docker compose exec mcp python -m pytest tests/unit -v`; integration equivalents where a workstream touches the worker), per project rules — the local Python environment lacks dependencies and DB access.

## Out of Scope

- **Any analysis tools** (correlation, trend, anomaly, hotspot, risk, narrative) — hard rule.
- **Any change to the routing rules themselves**: Harmony-first, one-whole-service-or-unpinned, never-union-across-services, OPeNDAP strictly as the worker's runtime fallback, direct-S3 only in-region+enabled. WS2 makes decisions visible; it changes none.
- **Hand-rolling any Harmony client behavior** — harmony-py remains the only Harmony client.
- **A structured `error_detail` column / Alembic migration on the jobs table.** WS3 deliberately uses the existing string column plus provenance-event detail; a typed error column can be its own PRD if the string convention proves insufficient.
- **Discovering OPeNDAP endpoints for collections that currently have none**, extending discovery to point-sample/non-grid-swath plans, or live per-DAAC validation beyond the existing opt-in live suite. WS1 contains the quirks we have; it does not hunt new ones.
- **The two already-fixed triage items** (GIBS authoritative lookup; job-not-found message) — closed, verify-only via their existing tests.
- **`cancel_retrieval`'s no-op-on-terminal behavior** — re-confirmed intentional twice already; untouched.
- **Simplifying or capping the *stored* AOI geometry** — the durable payload keeps full fidelity; only the tool response slims. Geometry simplification (e.g., for a future shape-subset payload limit) is separate work.
- **Changing any MCP tool's name or required parameters.** WS4 adds optional parameters and slims response payloads; nothing breaks an existing call.
