# PRD: Move OPeNDAP planning behind its own seam and deepen typed handle resolvers

> Triage label to apply when published: `ready-for-agent`

## Problem Statement

Two unrelated but similarly-shaped deepening opportunities showed up in the same architecture review, after the durable request spec (`RequestSpec` + `providers.build`) had already been given a type:

**OPeNDAP planning leaks into the generic retrieval tool.** `_submit_retrieval` in `tools/retrieval.py` is a ~120-line function that does resolve-handles, fetch-capabilities, route, and persist — but it *also* reaches into `providers/opendap.py`'s private, underscore-prefixed functions (`_discover_grid_geometry`, `_resolve_from_cmr`) to do OPeNDAP-specific discovery work: granule URL discovery, coordinate-name resolution, and grid-geometry/hyperslab computation. The OPeNDAP-only steps are interleaved with the generic planning steps inside one function, and the boundary between "retrieval planning" and "OPeNDAP" is drawn through the middle of that function rather than at a module edge. From a maintainer's perspective: "I have to open `opendap.py`'s private internals to understand what `retrieval.py` is doing, and any change to OPeNDAP's discovery logic risks breaking an import contract that isn't actually a public interface."

**The same handle-resolution block is copy-pasted six times.** `coverage.py`, `tools/retrieval.py`, `transform.py`, `preview.py`, `provenance.py`, and `understanding.py` (`describe_dataset`) each re-implement their own version of "type-check the handle prefix → resolve within the workspace (the cross-workspace isolation gate) → pull a field out of the payload." The shapes differ slightly (some resolve `dataset_` → `concept_id`, some resolve `aoi_` → bbox, some resolve `obs_`/`cube_` → `storage_key` + `media_type`), but the pattern and the isolation gate they all depend on is identical. A change to that isolation gate, or to how a malformed payload is reported, has to be made — and tested — in six places instead of one. From a maintainer's perspective: "I fixed the cross-workspace check in one tool and then found the same bug live in five others."

## Solution

**OPeNDAP planning:** expose `opendap.plan_subset(cmr, caps, concept_id, bbox, time_range, variables)` as `providers/opendap.py`'s public planning entry point. It owns granule-URL discovery (moved in from `tools/retrieval.py`'s `_discover_opendap_urls`/`_opendap_url_of`), coordinate-name resolution, and grid-geometry/hyperslab discovery — everything currently split between `retrieval.py`'s private imports and `opendap.py`'s already-private helpers. `_submit_retrieval` calls `plan_subset(...)` once and stops importing any `_`-prefixed symbol from `opendap.py`. The OPeNDAP module gets its own test surface instead of being exercised only indirectly through the retrieval tool's tests.

**Typed handle resolvers:** deepen the workspace layer with three typed resolver functions — `resolve_dataset(workspace_id, handle) → concept_id`, `resolve_aoi(workspace_id, handle) → bbox`, and `resolve_materialized(workspace_id, handle) → (storage_key, media_type, payload)` — each doing the type-check → isolation-gated lookup → payload-extraction sequence exactly once. The six tool modules become thin callers of these resolvers instead of each re-implementing the sequence.

These two changes are independent (different modules, different call sites) and can land in either order or in parallel.

## User Stories

1. As a maintainer, I want `tools/retrieval.py` to stop importing underscore-prefixed symbols from `providers/opendap.py`, so that the module boundary I can see in the import statements is the module boundary that actually exists.
2. As a maintainer, I want all OPeNDAP-specific planning logic (granule URL discovery, coordinate-name resolution, grid-geometry discovery) to live inside `providers/opendap.py`, so that I can understand and change OPeNDAP's planning behavior by reading one file.
3. As a maintainer, I want `opendap.plan_subset(...)` to be the single entry point the retrieval tool calls for OPeNDAP planning, so that adding a new discovery step to OPeNDAP touches one module instead of two.
4. As a maintainer, I want `_submit_retrieval` to shrink to its generic responsibilities (resolve handles, fetch capabilities, choose format, build the plan, route, mint handles, persist, record provenance, enqueue), so that the function is readable as "the shared planning core," not "planning core plus a nested OPeNDAP planner."
5. As a maintainer, I want `opendap.plan_subset` to have its own unit tests independent of `tools/retrieval.py`'s tests, so that a regression in OPeNDAP discovery is caught at the OPeNDAP seam, not only downstream in a retrieval integration test.
6. As an Earthdata MCP user, I want the behavior of `retrieve_data`/`retrieve_subset`/`retrieve_timeseries` for gridded and swath collections with an OPeNDAP fallback to be unchanged by this refactor, so that moving code around introduces no functional regression (same granule URLs discovered, same coordinate names resolved, same grid geometry computed, same `RequestSpec` fields populated).
7. As an Earthdata MCP user, I want a request for a collection with no OPeNDAP endpoints, or a point-sample request, to continue skipping OPeNDAP discovery entirely (no `plan_subset` call), so that the refactor doesn't add discovery overhead to requests that never needed it.
8. As a maintainer, I want the durable request spec (`RequestSpec.from_plan`) to keep receiving `opendap_urls`, `coord_lat`, `coord_lon`, `lat_axis`, `lon_axis`, and `var_dims` exactly as it does today, so that `providers.build` and job resumption continue to work unchanged — this PRD changes where the discovery code lives, not the spec's shape or the worker's runtime-fallback contract.
9. As a maintainer, I want a resolver that type-checks a `dataset_` handle, resolves it within the workspace (denying cross-workspace access), and returns its `concept_id`, so that every tool that needs a dataset's `concept_id` calls one function instead of re-implementing the sequence.
10. As a maintainer, I want a resolver that type-checks an `aoi_` handle, resolves it within the workspace, and returns its bbox as a `(west, south, east, north)` tuple of floats, so that every tool that needs an AOI's bbox calls one function instead of re-implementing the sequence.
11. As a maintainer, I want a resolver that type-checks an `obs_`/`cube_` handle, resolves it within the workspace, and returns `(storage_key, media_type, payload)` after confirming the handle is a materialized (`status == "ready"`) result, so that every tool that reads a materialized result calls one function instead of re-implementing the sequence.
12. As a maintainer, I want `coverage.py`'s `_resolve_handles` (dataset + aoi → concept_id + bbox string) to be rebuilt from the two typed resolvers rather than hand-rolled, so that CMR's `bounding_box` string format is derived in one place from the same bbox tuple every other caller uses.
13. As a maintainer, I want `tools/retrieval.py`'s `_resolve_handles` to be rebuilt from the two typed resolvers, so that the planning tool and the coverage tools share the exact same handle-resolution behavior instead of two independently-maintained near-duplicates.
14. As a maintainer, I want `preview.py`'s `_aoi_bbox` and `transform.py`'s `_aoi_bbox` (currently byte-for-byte identical private functions) collapsed into the one shared `resolve_aoi` resolver, so that the duplication is deleted rather than kept as a third and fourth copy.
15. As a maintainer, I want `transform.py`'s `_load_source` and `preview.py`'s materialized-handle lookups (in `summarize_dataset` and `inspect_statistics`) to use the shared `resolve_materialized` resolver, so that the "is this handle actually a ready, materialized result" check and its error message are defined once.
16. As a maintainer, I want `provenance.py`'s `cite_dataset` and `understanding.py`'s `describe_dataset` to use the shared `resolve_dataset` resolver for their `concept_id` lookup, so that every dataset-handle consumer shares one implementation of "missing `concept_id`" error handling.
17. As an Earthdata MCP user, I want a wrong-typed handle (e.g. passing a `job_` handle where a `dataset_` handle is expected) to keep raising the same `ValueError` message shape it does today across every one of the six tools, so that this refactor is invisible to me as a caller.
18. As an Earthdata MCP user, I want a cross-workspace handle access to keep being denied with `CrossWorkspaceError` across every one of the six tools, so that the isolation guarantee (CLAUDE.md: cross-workspace access is denied, not silently served) is preserved exactly, not weakened by consolidation.
19. As a maintainer, I want the resolvers' isolation-gate behavior (`CrossWorkspaceError`/`HandleNotFoundError` propagation) to be tested once at the resolver seam, so that I don't need six near-identical cross-workspace tests to trust the guarantee holds everywhere it's used.
20. As a maintainer, I want each of the six tool files to shrink by the amount of resolution logic it no longer owns, so that the files read as "the tool's actual logic" rather than "the tool's logic plus a private handle-resolution helper."
21. As a maintainer extending the MCP server with a new tool that needs a dataset's `concept_id` or an AOI's bbox, I want to import the shared resolver rather than write a seventh copy of the same block, so that the duplication doesn't grow with each new tool.

## Implementation Decisions

### OPeNDAP planning seam

- **New public entry point.** `providers/opendap.py` gains `plan_subset(cmr, caps, concept_id, bbox, time_range, variables) → OpendapPlan` (or equivalent return shape carrying `opendap_urls`, `coord_lat`, `coord_lon`, `lat_axis`, `lon_axis`, `var_dims`) as its public planning face. Internally it composes the granule-URL discovery (moved in from `retrieval.py`), `_resolve_from_cmr`, and `_discover_grid_geometry` — all of which may stay private *within* `opendap.py` now that nothing outside the module calls them directly.
- **Retire the private cross-module imports.** `tools/retrieval.py` stops importing `_discover_grid_geometry` and `_resolve_from_cmr` from `providers/opendap.py`. `_discover_opendap_urls` and `_opendap_url_of` move out of `retrieval.py` and into `opendap.py` as the first stage of `plan_subset`.
- **Same call-site conditions, same fail-soft posture.** `plan_subset` is invoked from `_submit_retrieval` under the same conditions as today's discovery block (`not needs_point_sample and caps.output_shape in ("grid", "swath") and bbox is not None`), and it preserves every existing fail-soft behavior: a CMR/network error still falls back to default coordinate names and unresolved variables; grid-geometry discovery still yields `(None, None, {})` on any failure or non-"grid" shape; an ambiguous variable name still raises outside the network-error handling so it is never swallowed.
- **`RequestSpec.from_plan` is unaffected.** The spec still receives `opendap_urls`, `coord_lat`, `coord_lon`, `lat_axis`, `lon_axis`, and `var_dims` — this PRD changes which module computes them, not what the durable spec stores or how `providers.build` reconstructs the provider from it.
- **Routing and the worker's runtime-fallback contract are unchanged.** Harmony is still tried first; OPeNDAP remains the worker's runtime fallback when a real Harmony submit fails, never a planning-time choice (CLAUDE.md hard rule). This PRD only relocates *where the OPeNDAP planning inputs are computed*, not when OPeNDAP is chosen.

### Typed handle resolvers

- **Three resolver functions**, added either as methods on `WorkspaceStore` or as free functions in a new small `handles.py` module that takes a `WorkspaceStore` and calls into it (either placement satisfies the seam; the concrete decision is left to whoever implements — reuse the existing `_resolve_handles`/`_aoi_bbox` call shapes as the contract to match):
  - `resolve_dataset(store, workspace_id, dataset_handle) → concept_id`: type-checks for `HandleType.DATASET`, resolves via `store.get_handle` (isolation gate), raises `ValueError` if `concept_id` is missing from the payload — matching today's error message shape in each of the six call sites.
  - `resolve_aoi(store, workspace_id, aoi_handle) → (west, south, east, north)`: type-checks for `HandleType.AOI`, resolves via `store.get_handle`, raises `ValueError` if `bbox` is missing or not length 4. Handles the `aoi_handle is None` case at the caller (some call sites treat a missing AOI as "no bbox," which is a caller decision, not the resolver's).
  - `resolve_materialized(store, workspace_id, handle) → (storage_key, media_type, payload)`: type-checks for `HandleType.OBS` or `HandleType.CUBE`, resolves via `store.get_handle`, raises `ValueError` if `status != "ready"` or `storage_key`/`media_type` is missing.
- **Every existing call site is rebuilt on top of these three, not replaced with new logic.** `coverage.py._resolve_handles`, `tools/retrieval.py._resolve_handles`, `preview.py._aoi_bbox`, `transform.py._aoi_bbox`, `transform.py._load_source`, `provenance.py.cite_dataset`'s inline resolution, and `understanding.py.describe_dataset`'s inline resolution all become thin compositions of `resolve_dataset`/`resolve_aoi`/`resolve_materialized`. Where a caller currently builds a CMR bbox string (`coverage.py`), that string-formatting step stays at the caller, applied to the tuple `resolve_aoi` returns — the resolver returns the typed value, not a CMR-specific string.
- **Error semantics are preserved exactly**, not just "close enough": the same exception types (`ValueError` for a wrong handle type or malformed payload, `CrossWorkspaceError`/`HandleNotFoundError` propagated unchanged from `store.get_handle`) and materially the same message content, since existing tests assert on these.
- **No change to `WorkspaceStore`'s existing `get_handle`/`put_handle`/`update_handle`/`list_handles` contract.** The resolvers are a layer on top, not a replacement for the store's isolation gate.

## Testing Decisions

- **What makes a good test here:** assert external, observable behavior at the public seam — for OPeNDAP planning, the `OpendapPlan`/tuple `plan_subset` returns for a given fake-CMR response and bbox/time/variables input; for the resolvers, the typed value (or raised exception) each resolver returns for a given fake `WorkspaceStore` state. Not the internal call sequence.
- **Seam 1 — `opendap.plan_subset` (new, highest seam for this candidate).** Test directly against `providers/opendap.py` with a fake CMR (stubbed `get_variables`/`search_granules`), replacing the current indirect coverage that exercises this logic only through `tools/retrieval.py`. Cover: granule URLs discovered from a fake CMR search; coordinate names and variable resolution (bare leaf → group path, already-qualified pass-through, zero-match pass-through, ambiguous-leaf raises); grid-geometry discovery for a "grid" shape and its absence for "swath"; the fail-soft fallback on a CMR error. Prior art: `tests/unit/test_providers/test_opendap.py` already covers `_constraint_expression` and can be extended with `plan_subset`-level tests using the same fake-CMR stubbing pattern used in `tests/unit/test_tools/test_retrieval.py`.
- **Seam 2 — `tools/retrieval.py`'s planning core, updated not duplicated.** The existing `_submit_retrieval` tests (e.g. `test_retrieve_data_stores_request_spec_not_url`, `test_retrieve_data_collects_all_window_granules_for_opendap_fallback`) continue to assert the same persisted-spec behavior; they should keep passing unchanged if `plan_subset` is a faithful extraction, since they test the tool's external behavior (what lands in the spec), not `opendap.py`'s internals.
- **Seam 3 — the three typed resolvers (new, highest seam for this candidate).** Test `resolve_dataset`/`resolve_aoi`/`resolve_materialized` directly against a fake/in-memory `WorkspaceStore` (or the existing test DB fixture, matching whatever `tests/unit/test_tools/test_coverage.py` and `test_retrieval.py` already use for `WorkspaceStore`). Cover, once each: correct type + correct workspace → typed value; wrong handle type → `ValueError`; cross-workspace access → `CrossWorkspaceError`; missing/malformed payload field → `ValueError`. Prior art: `test_wrong_dataset_handle_type_raises`, `test_wrong_aoi_handle_type_raises`, and `test_cross_workspace_denied` in `tests/unit/test_tools/test_coverage.py`; the equivalent tests in `test_preview.py`, `test_transform.py`, `test_provenance.py`, and `test_retrieval.py`.
- **Seam 4 — each of the six tool modules keeps a thin regression test.** Each tool's existing wrong-handle-type / cross-workspace test should still pass after the tool is rebuilt on the shared resolvers — these become integration checks that the tool wires the resolver correctly, not re-tests of the isolation gate itself. Do not delete these tests; keep them as the "the tool actually uses the resolver" check, but don't add new ones beyond what already exists per tool.
- **No new seams beyond these four.** They test genuinely distinct things (OPeNDAP discovery logic, the retrieval tool's persisted output, the resolvers' isolation/type/payload behavior, and each tool's wiring) and are not collapsed further.
- **Run via the project's Docker test workflow** (`docker compose exec mcp python -m pytest tests/unit -v`), per project rules — the local Python environment lacks dependencies and DB access.

## Out of Scope

- Candidate 05 from the same architecture review (narrowing `providers/router.py`'s dead branches) — the review itself flags this as requiring a grill session before any work starts, since CLAUDE.md names `router.py` routing canon and the "dead" branches may be deliberate structural fallback insurance. Not part of this PRD.
- Any change to Harmony-first routing, the union-trap handling, or when OPeNDAP fires as the worker's runtime fallback. This PRD relocates planning-time discovery code; it does not change any routing decision.
- Any change to `RequestSpec`'s shape, `to_jsonb`/`from_jsonb`, or `providers.build` — these were already deepened in the prior `RequestSpec` + `providers.build` PRD/commit and are not reopened here.
- Adding new resolver methods beyond the three named (`resolve_dataset`, `resolve_aoi`, `resolve_materialized`) — e.g. a `job_`/`obs_` pending-status resolver for `get_retrieval_status`/`cancel_retrieval` in `tools/retrieval.py` — since those call sites don't fit the dataset/aoi/materialized shapes and weren't identified as duplicated in the review.
- Any change to the six tools' public signatures, return shapes, or MCP-facing behavior. This is an internal deepening; callers of `check_coverage`, `retrieve_data`, `subset`, `preview_dataset`, `get_provenance`, `describe_dataset`, etc. should observe no difference.
- Performance work (e.g. batching CMR calls across resolvers) beyond preserving the existing "one `get_variables` call per collection" posture already established for OPeNDAP discovery.
